📁 FILE REFERENCE
─────────────────

combined_simple_teleop_real_logger.py (520 lines)
  Main script: live teleoperation + HDF5 logging + camera capture
  Logs: arm/{raw_pose, safe_pose, smoothed_pose}, hand/*, camera/*, timestamps
  Usage: python combined_simple_teleop_real_logger.py [options]
  Options: --enable-camera, --camera-width, --camera-height, --camera-fps, etc.
  Default: records with HDF5 logging enabled; camera optional

robot_pose_controller.py (420 lines)
  Controller class: send any 6D pose + hand commands  
  Usage: python robot_pose_controller.py --pose X Y Z Rx Ry Rz
  Also imported by replay_hdf5.py

replay_hdf5.py (370 lines)
  Replay recorded trajectories from HDF5 with optional video playback
  Loads: arm/smoothed_pose, hand/leap_pose, timestamps, camera/color (optional)
  Usage: python replay_hdf5.py file.hdf5 [options]
  Options: --speed, --show-video, --dry-run
  Features: homing phase, smooth motion, timing preservation, video sync

verify_sync.py (150 lines)
  Verify sync between camera video and arm trajectory in HDF5 files
  Visualizes: camera frames side-by-side with trajectory plot
  Controls: Space=pause, Q=quit, arrows=scrub
  Usage: python verify_sync.py logs/teleop_*.hdf5 [--speed MULTIPLIER]

track.py (200+ lines)
  Vive Tracker module for real-time tracking via OpenVR
  Used by: combined_simple_teleop_real_logger.py
  Provides: 4x4 transformation matrices, calibration, live tracker polling

teleoperate.py (300+ lines)
  Core ViveToRMMapper class for tracker→robot coordinate transforms
  Implements: 6-step safety filter, pose smoothing, hold/move logic
  Used by: combined_simple_teleop_real_logger.py, robot_pose_controller.py

run_with_manus.sh
  Convenience script to launch MANUS SDK and teleoperation together
  Usage: ./run_with_manus.sh
  Handles: virtualenv detection, SDK startup, HOME env for SteamVR

leap_hand_utils/
  Python utilities for LEAP Hand control
  ├─ dynamixel_client.py: Dynamixel motor protocol
  ├─ leap_hand_utils.py: joint/motor conversions

logs/
  Directory: HDF5 recordings (auto-created)
  Format: teleop_YYYYMMDD_HHMMSS.hdf5
  Size: ~5-10 MB/min without camera, ~100-200 MB/min with camera (compressed)

🚀 COMMAND QUICK REFERENCE
───────────────────────────

Record (without camera):
  python combined_simple_teleop_real_logger.py

Record (with camera):
  python combined_simple_teleop_real_logger.py --enable-camera --camera-width 640 --camera-height 480 --camera-fps 30

Replay (real-time):
  python replay_hdf5.py logs/teleop_*.hdf5

Replay with video (real-time):
  python replay_hdf5.py logs/teleop_*.hdf5 --show-video

Replay (half speed):
  python replay_hdf5.py logs/teleop_*.hdf5 --speed 0.5

Verify camera/arm sync in file:
  python verify_sync.py logs/teleop_*.hdf5

Inspect file info (dry-run):
  python replay_hdf5.py logs/teleop_*.hdf5 --dry-run

Send single pose:
  python robot_pose_controller.py --pose 0.3 0.0 0.2 0.0 0.0 0.0

Convenient launch (with MANUS):
  ./run_with_manus.sh

Full help:
  python combined_simple_teleop_real_logger.py --help
  python replay_hdf5.py --help
  python robot_pose_controller.py --help
  python verify_sync.py --help


📊 FOR RESEARCHERS
───────────────────

To understand the exact data flow:
  1. Read DATA_PIPELINE.md systematically from top to bottom
  2. Look at code locations cited for each stage
  3. Refer to reference tables for data formats
  4. Use diagrams to explain to others

To reproduce results:
  1. Check SETUP_AND_EXECUTION.md for all setup details
  2. Follow "Pre-Execution Checklist" exactly
  3. Use HDF5 inspection to verify data quality
  4. Reference troubleshooting section for issues

To train a diffusion or learning-based policy:
  1. Record demonstrations with --enable-camera for context
  2. Extract arm/raw_pose for unfiltered human intent (preserves corrections)
  3. Extract camera/color for visual observations
  4. Optionally use arm/safe_pose or arm/smoothed_pose for post-filtering
  5. Use timestamp alignment for synchronization
  6. See DATA_PIPELINE.md Stage 1B for raw_pose definition

To compare with other systems:
  Data_PIPELINE.md shows exact processing stages, so you can:
  - Identify which stages are novel to this system
  - Compare safety approaches (6-step filter documented)
  - Evaluate logging completeness (raw, safe, smoothed arm data)
  - Validate replay fidelity against recorded video
  - Assess data density: all fields logged at control Hz (~125 Hz typical)
